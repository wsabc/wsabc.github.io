蓝绿发布(BlueGreenDeployment)

示意图

意图

  1. 关注自动化部署的挑战: 从final stage到production的切换
  2. 减少downtime

原理(操作)

  1. 准备两套生产环境,尽可能一样(比如硬件就可以不同)
  2. 任意时刻,其中一个环境(如blue)是作为生产环境存在
  3. 准备新版本的时候,在另外一个环境(如green)上进行final stage testing
  4. 一旦达到验收标准,将所有请求从blue切换到green

优缺点

  1. 可以快速回滚,一旦发现错误,可以立即从green切回到blue

  2. 一旦环境稳定,就可以将blue作为下一次final stage环境,这样在blue-green-blue中循环下去

  3. 类似热备(Host-Standby)的工作,其实每次都是一次灾备(disaster-recovery)演练

  4. 可以灵活运用,基础原理就是在两个易于切换的环境中切换.比如可以在Web Server中切换,也可以在业务层切换

  5. 缺点:一旦发生回滚,试运行期间(green)上的新进数据如何保持一致性.有几种方法(但是会提高系统复杂度)

    a. 原则:如何处理数据,是根据系统的设计而定

    b. 方法1: 同时向两个环境发送transaction,使得blue是green的一个备份

    c. 方法2: 切换之前,限制green为read-only模式,通过观察周期之后,再切换到read-write模式

挑战

数据库是使用这种技术面临的最常见的挑战之一,尤其是有Schema变更的时候.应对策略:

将数据库的变化从应用的变化中抽离出来

  1. 重构数据库,同时支持新版本和老版本,部署,观察,同时设置回滚点以便rollback
  2. 部署新版本应用
  3. 所有都稳定之后,移除数据库对老版本的支持

—End
