HTTP协议

发表于 2019-08-26 | 更新于 2019-09-30 | 分类于 http | 阅读次数：

HTTP是最常见的应用层协议,也是WEB开发不可避免的要熟悉的知识.

HTTP简介

HTTP(超文本传输协议)是在网络上传输HTML的协议,用于浏览器和服务器之间的通信,由请求和响应组成,是一个标准的CS模型.同时HTTP是一个无状态协议.

HTTP协议通常承载于TCP协议之上,有时也承载于TLS或SSL协议层之上,这个时候HTTP就成了我们常说的HTTPS.

HTTP最广泛使用的版本是1.1;目前最新的版本是2.0.之前还有0.9,1.0.

HTTP 1.1相对于1.0最主要的变化是支持长连接(Persistent Connection:建立一次 TCP 连接就能进行多次 HTTP 通信)和请求的流水线(Pipelining:在同一条长连接上连续发出请求,而不用等待响应返回,这样可以减少延迟)处理.在一个TCP连接上可以传送多个HTTP请求和响应.减少了建立和关闭连接的消耗和延迟.在HTTP1.1中默认开启Connection:keep-alive.

HTTP2.0多路复用(多个请求可同时在一个连接上并行执行)相对于HTTP1.x的管道流水线(若干个请求排队串行化处理)来说,性能更好.

HTTP 方法

  • CONNECT: 浏览器需要先通过HTTP明文形式向代理服务器发送一个CONNECT请求告诉它目标站点地址及端口号.当代理服务器收到这个请求后,会在对应的端口上与目标站点建立一个TCP连接,连接建立成功后返回一个HTTP 200状态码告诉浏览器与该站点的加密通道已建成.接下来代理服务器仅仅是来回传输浏览器与该服务器之间的加密数据包,代理服务器并不需要解析这些内容以保证HTTPS的安全性.因此只有当浏览器配置为使用代理服务器时才会用到CONNECT方法.

  • HEAD: 获取报文首部,和GET类似,但是不包含BODY.主要用于确认 URL 的有效性以及资源更新的日期时间等.

  • OPTIONS: 查询指定URL支持的方法

  • GET: 获取资源

  • POST: 传输数据,创建资源

  • PUT: 传输数据,更新整个资源

  • PATCH: 传输数据,更新资源部分属性

  • DELETE: 删除资源

  • TRACE: 服务器会将通信路径返回给客户端,通常不会使用 TRACE,它容易受到 XST 攻击(Cross-Site Tracing,跨站追踪).

HTTP 状态码

状态码 类别 典型状态码
1XX Informational(信息性状态码) 100 Continue
2XX Success(成功状态码) 200, 201, 204, 206 Partial Content
3XX Redirection(重定向状态码) 301, 302, 304, 307
4XX Client Error(客户端错误状态码) 400, 401, 403, 404
5XX Server Error(服务器错误状态码) 500, 503

常见HTTP Header

Header名字 位置 含义
Connection 通用 控制不再转发给代理的首部字段,管理持久连接
Transfer-Encoding 通用 指定报文主体的传输编码方式
Authorization 请求 Web 认证信息
User-Agent 请求 HTTP 客户端程序的信息
Location 响应 令客户端重定向至指定 URI
Content-Length 通用 实体主体的大小
Content-Type 通用 实体主体的媒体类型

URI vs. URL

URI, 统一资源标识符, 用来唯一的标识一个资源.
URL,统一资源定位器, 它是一种具体的URI,即可以用来标识一个资源,还指明了如何定位到这个资源.
URN,统一资源命名,是通过名字来标识资源,比如urn:isbn:1234567890

URI是以一种抽象的高层次概念定义统一资源标识,而URL和URN则是具体的资源标识的方式.简单来说每个 URL 都是 URI,但不一定每个 URI 都是 URL.

Cookie vs. Session

HTTP/1.1 引入 Cookie 来保存状态信息.Cookie 是服务器发送到用户浏览器并保存在本地的一小块数据,它会在浏览器之后向同一服务器再次发起请求时被携带上,用于告知服务端两个请求是否来自同一浏览器.

新的浏览器 API 已经允许开发者直接将数据存储到本地,如使用LocalStorage 或 IndexedDB.

没有指定Expires或者max-age的 cookie 是会话期cookie,浏览器关闭后自动删除.指定了这两个属性之一的就变成持久性 cookie.

Domain 标识指定了哪些主机可以接受 Cookie,如果不指定,默认为当前文档的主机–不包含子域名;如果指定了 Domain,则一般包含子域名.

Path 标识指定了主机下的哪些路径可以接受 Cookie–该 URL 路径必须存在于请求 URL 中,以字符/作为路径分隔符,子路径也会被匹配.

标记为 HttpOnly 的 Cookie 不能被 JavaScript 脚本调用,因此使用 HttpOnly 标记可以在一定程度上避免 XSS 攻击.

标记为 Secure 的 Cookie 只能通过被 HTTPS 协议加密过的请求发送给服务端.

Session 是存储在服务器上的文件,数据库或者内存中.通过 Cookie 中的 SessionID关联.如果浏览器禁用了 cookie,就只能使用 session,并且需要使用 URL 重写技术,将 sessionId 放到 URL 中进行传递.

缓存

一般有 2 种方法: 一是代理服务器进行缓存;而是客户端浏览器进行缓存.HTTP/1.1 通过 Cache-Control 首部字段来控制缓存.

多线程下载和断点续传

断点续传/下载: 在网络情况不好的时候,可以在断开连接以后,仅继续获取部分内容.

多线程下载: 对大型文件,开启多个线程,每个线程下载其中的某一段,最后下载完成之后,在本地拼接成一个完整的文件.

HTTP/1.1 中,响应头部 Accept-Ranges 用来标记是否支持范围请求,它只有一个可选参数 bytes.

一旦服务器端文件发生变化,需要使用If-Range指定Etag或者Last-Modified的值去判断是否更新.此时,如果两次操作的都是同一个资源文件,就会继续返回 206 状态码,开始后续的操作,反之则会返回 200 状态码,表示文件发生改变,要从头下载.

分片上传

分片上传功能支持将一个文件切割为一系列特定大小的小数据片,分别将这些小数据片分别上传到服务端,全部上传完后再在服务端将这些小数据片合并成为一个资源.每片上传完都有一个标记,客户端需要持久化这些标记,一旦发生断连,恢复的时候可以选择性的续传.类似 TCP 过程.

HTTP Auth

Basic

HTTP 摘要认证(Digest Access)

基本思路是请求方把用户名口令和域做一个MD5 MD5(username:realm:password) 然后传给服务器.根据qop和nonce做不同的复杂化处理.算法虽然繁琐,但是关键是用户的 password 是否复杂,容易暴力破解,还容易收到中间人攻击.

App Secret Key + HMAC

MAC(Message Authentication Code)是一种用于给消息签名的技术,防篡改.HMAC(Hash-based Authenticsation Code)是指利用 Hash 技术完成这一过程,如 SHA-256.

把AppID和HMAC用于API认证,目前来说最专业的应该是AWS了.目前没有统一标准,各家实现不一样.

JWT

JWT是一个比较标准的认证解决方案,在Java圈里应该用的是非常普遍的.JWT签名也是一种MAC方法.

OAuth 1.0

流程有三方:User, Consumer 和 Service Provide,所以又叫 3-legged flow,三脚流程.

OAuth 1.0 也有不需要用户参与的,只有Consumer 和 Service Provider 的, 也就是 2-legged flow 两脚流程,其中省掉了用户认证的事.

OAuth 2.0

从Digest Access,到AppID+HMAC,再到JWT,再到OAuth 1.0,这些个API认证都是要向Client发一个密钥然后用HASH或是RSA来签HTTP的请求,这其中有个主要的原因是,以前的HTTP是明文传输,所以在传输过程中很容易被篡改,于是才搞出来一套的安全签名机制,这些个认证的玩法是可以在HTTP明文协议下玩的.

OAuth 2.0依赖于TLS/SSL的链路加密技术(HTTPS),完全放弃了签名的方式,认证服务器再也不返回什么 token secret 的密钥了,所以OAuth 2.0是完全不同于1.0 的,也是不兼容的.目前,Facebook 的 Graph API 只支持OAuth 2.0协议,Google 和 Microsoft Azure 也支持Auth 2.0,国内的微信和支付宝也支持使用OAuth 2.0.

Authorization Code Flow, 这个是 3 legged 的,适用于用户给第三方应用授权访问自己信息的场景.

Client Credential Flow,这个是 2 legged 的,是一个简化版的API认证,主要是用于认证服务器到服务器的调用,也就是没有用户参与的的认证流程.

不要使用GET的方式在URL中放入secret之类的东西,因为很多proxy或gateway的软件会把整个URL记在Access Log文件中.

CORS(Cross-Origin Resource Sharing)

CORS是指一个 web 程序想访问另一个 origin 里面的资源.domain,protocol,port 有一个不同都算是另一个 origin.因为安全原因,浏览器会限制脚本里的 cors 请求.比如XMLHttpRequest.想要跨域访问,需要对方服务器配合,返回相应的 Http Header.如Access-Control-Allow-Origin: *等.

Preflighted requests是指浏览器在执行真正的跨域请求之前,发送一个 OPTIONS 请求探测一下.

解决跨域的办法:

  1. JSOP,几乎不再使用了.只支持 GET 请求.JSONP的优势在于支持老式浏览器,以及可以向不支持CORS的网站请求数据.
  2. CORS,支持所有类型的HTTP请求.具体参考跨域资源共享 CORS 详解
  3. 代理服务器.一般 web 开发前后分离并且部署在不同域时,前端都会使用代理访问,如 nodejs.

HTTP长连接

TCP Keep Alive 和 HTTP Keep Alive 是两个目的不同的技术,不存在谁依赖于谁的关系.TCP Keep Alive 用于探测对端是否存在,而 HTTP Keep Alive 用于协商以复用 TCP 连接.即便一个 TCP 连接未启用 Keep Alive 功能,也不妨碍 HTTP 层面开启长连接.

当需要建立 HTTP 长连接时,HTTP 请求头应该包含Connection: Keep-Alive, 同样如果服务端同意建立长连接,那么响应头也应该包含Connection: Keep-Alive,关闭的时候使用Connection: Close.

HTTP方法幂等性

幂等就是值不变性,相同的请求得到相同的响应结果,不会出现相同的请求出现不同的响应结果.

方法GET，HEAD，PUT，DELETE都有这种性质.同样,方法OPTIONS和TRACE不应该有副作用,因此具有内在的等幂性.

Delete有副作用,但是应该满足幂等性.POST所对应的URI并非创建的资源本身,而是资源的接收者.两次 POST 将会创建两个不同的实体,因此不具有幂等性.PUT 因为是同一个资源,应该具有幂等性.

WebSocket

是一种网络通信协议.因为 HTTP 协议有一个缺陷:通信只能由客户端发起,所以需要一个能”推送”的协议.

WebSocket 2008 年出现,2011 年成为国际标准, 它的最大特点就是,服务器可以主动向客户端推送信息,客户端也可以主动向服务器发送信息,是真正的双向平等对话,属于服务器推送技术的一种.

  1. 建立在 TCP 协议之上，服务器端的实现比较容易
  2. 与 HTTP 协议有着良好的兼容性。默认端口也是80和443，并且握手阶段采用 HTTP 协议，因此握手时不容易屏蔽，能通过各种 HTTP 代理服务器
  3. 数据格式比较轻量，性能开销小，通信高效;可以发送文本，也可以发送二进制数据
  4. 没有同源限制，客户端可以与任意服务器通信
  5. 协议标识符是ws（如果加密，则为wss），服务器网址就是 URL。如ws://example.com:80/some/path

参考WebSocket 教程 以及 websocketd

HTTPS

HTTPS 并不是新协议，而是让 HTTP 先和 SSL（Secure Sockets Layer）通信，再由 SSL 和 TCP 通信，也就是说 HTTPS 使用了隧道进行通信。

对称加密: 快,密钥不安全.

非对称加密: 慢, 公钥加密,私钥解密;私钥签名,公钥验签.密钥相对安全.

HTTPS 采用混合的加密机制，使用非对称密钥加密用于传输对称密钥来保证传输过程的安全性，之后使用对称密钥加密进行通信来保证通信过程的效率。

SSL/TLS

SSL是基于HTTP之下TCP之上的一个协议层,在SSL更新到3.0时,IETF对SSL3.0进行了标准化,并添加了少数机制(但是几乎和SSL3.0无差异),标准化后的IETF更名为TLS1.0(Transport Layer Security 安全传输层协议),可以说TLS就是SSL的新版本3.1.

目前TLS1.2 是最新的 SSL/TLS版本.TLS1.3即将到来.

SSL/TLS是为了解决互联网通信三大问题:

  1. 防窃听: 防止第三方获知通信内容 — 加密传输
  2. 防篡改: 防止第三方篡改通信内容 — 校验机制
  3. 防冒充: 防止第三方身份冒充 — 身份证书(将公钥放在数字证书中,只要证书是可信的,公钥就是可信的)

SSL/TLS协议的基本思路是采用公钥加密法,也就是说,客户端先向服务器端索要公钥,然后用公钥加密信息,服务器收到密文后,用自己的私钥解密.

SSL 四次握手

  1. 客户端（通常是浏览器）先向服务器发出加密通信的请求，这被叫做ClientHello请求。

    包含: TLS 版本, client nonce, 支持加密算法列表, 支持压缩算法列表

  2. 服务器回应（SeverHello）

    确认TLS 版本(版本不一致,服务器关闭加密通信); server nonce; 确认加密算法; 服务器证书.

    如果需要客户端证书,还会有 CertificateRequest.

  3. 客户端回应

    验证证书; pre-master key(用公钥加密); 编码改变通知(协商的加密算法和密钥); Finished.

  4. 服务器的最后回应

    收到 pre-master key 计算 session key;编码改变通知(协商的加密算法和密钥); Finished.

为什么需要三个随机数通过一个密钥导出器最终导出一个对称密钥?

pre master key的存在在于SSL协议不信任每个主机都能产生完全随机的随机数,如果随机数不随机,那么pre master secret就有可能被猜出来,那么仅适用pre master secret作为密钥就不合适了,因此必须引入新的随机因素,那么客户端和服务器加上pre master key三个随机数一同生成的密钥就不容易被猜出了,一个伪随机可能完全不随机,可是是三个伪随机就十分接近随机了.

Keyless服务,即你把网站放到它们的CDN上,不用提供自己的私钥,也能使用SSL加密链接.

私钥的作用主要是解密 pre-master key.这一步让客户参与就可以了.

如果有人窃听通信，他可以知道双方选择的加密方法，以及三个随机数中的两个。整个通话的安全，只取决于第三个随机数（Premaster secret）能不能被破解。把握手阶段的算法从默认的RSA算法，改为DH[Diffie-Hellman)算法,Premaster secret不需要传递，双方只要交换各自的参数，就可以算出这个随机数.这样就提高了安全性。

抓包工具

常见的抓包工具有：Charles，Fiddler，WireShark。具体用法参照官网。

HTTP/2

HTTP/2 是一个彻底的二进制协议，头信息和数据体都是二进制，并且统称为”帧”（frame）：头信息帧和数据帧。协议的主要目的是提高网页性能。

HTTP/2 复用TCP连接，在一个连接里，客户端和浏览器都可以同时发送多个请求或回应，而且不用按照顺序一一对应，这样就避免了”队头堵塞”。这样双向的、实时的通信，就叫做多工（Multiplexing）。

服务器推送（server push）是 HTTP/2 协议里面，唯一一个需要开发者自己配置的功能。其他功能都是服务器和浏览器自动实现，不需要开发者关心。

常见攻击

XSS（跨站脚本攻击 - Cross-site scripting）

攻击者在网页上发布包含攻击性代码的数据。当浏览者看到此网页时，特定的脚本就会以浏览者用户的身份和权限来执行。

CSRF（跨站请求伪造攻击 - Cross-site request forgery）

攻击者通过各种方法伪造一个请求，模仿用户提交表单的行为，从而达到修改用户的数据，或者执行特定任务的目的。为了 假冒用户的身份，CSRF攻击常常和XSS攻击配合起来做。
