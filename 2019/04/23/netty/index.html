Netty

发表于 2019-04-23 | 更新于 2019-09-30 | 分类于 netty | 阅读次数：

目前Netty的稳定版是4.1.38.

Netty关键字: 异步 事件驱动 网络编程框架 快速开发 高性能 高扩展性 服务器端 客户端

入门

  1. Inbound入站,outbound出站.ChannelInboundHandler提供一个接口,抽象了入站的各种操作.ChannelInboundHandlerAdapter提供了一个基类,方便实现.
  2. ByteBuf是Netty里的基本缓冲类型,需要调用release方法释放.这个是handler的责任.可以通过ChannelHandlerContext分配空间.ByteBuf操作不需要flip,因为Netty维护2个指针,一个读,一个写.
  3. exceptionCaught异常发生时,通常是需要记日志和关闭对应的channel.有时候也可以回复客户端错误然后再关闭.
  4. channelRead当数据到来时就会调用这个方法.
  5. channelActive当连接建立并准备进行通信时调用这个方法.
  6. handlerAdded,handlerRemoved可以添加一些轻量级的初始化/释放工作,但是不能阻塞太长时间.
  7. NioEventLoopGroup是基于NIO selector的多线程接收器.boss用来接收连接类似mainReactor,worker用来处理建立的连接类似subReactor.线程池的大小可以通过构造方法或者具体实现配置.
  8. ServerBootstrap是一个助手类,用来简化构造服务器的代码.调用链一般是b.group.channel.childHandler.childOption.option,最后是bind,close.
  9. childHandler用来注册客户端连接的处理器.通过ChannelPipeline来构造通道处理路线.每个通道有自己的pipeline,并且是通道建立的时候就创建好了.pipeline的执行顺序(12345)是:inbound从1到5;outbound是从5到1.addLast则是按照顺序添加handler.
  10. option用来配置ServerSocketChannel,childOption用来配置SocketChannel.SO_BACKLOG用来配置server端accept对列大小,取决于min(backlog, somaxconn),超过会导致ECONNREFUSED错误.
  11. ChannelHandlerContext提供了各种IO操作和触发器.如write,flush,close等.
  12. Netty里的所有操作都是异步的.如果有先后关系的操作,可能需要放在future的回调里,比如close可能会在flush之前完成.
  13. Bootstrap用于构建客户端连接程序,只需要一个workergroup.调用链是b.group.channel.option.handler.connect.注意没有child*方法了(因为没有ServerSocketChannel这个parent了).
  14. 流式传输协议(如TCP)有一个数据包碎片化的问题,需要程序来组包.Netty提供了Decoder/Encoder功能(也是ChannelInbound/OutboundHandler实例).他们内部会维持一个累积的buffer用来存储数据.常见的有ByteToMessageDecoder,ReplayingDecoder等.
  15. 尽管ByteBuf解决了组包问题,但是不能复用,不易于理解.Netty支持在Encoder/Decoder中直接使用POJO的能力.
  16. 使用shutdownGracefully关闭eventloopgroup即可关闭应用,同时也会关闭该group的通道.
  17. Netty提供了丰富的编解码器,如LineBasedFrameDecoder,也可以自定义如ByteToMessageDecoder.

Netty作为类库使用

  1. 良好的Buffer API: 易用, 可扩展, 高效.
  2. 可监控的Future和事件循环: EventExecutorGroup, GlobalEventExecutor.
  3. 其他的辅助类: ThreadLocalPool, Extensible Enum, Attribute Map, Hashed Wheel Timer.

主要线程模型

传统IO 模型: 一个连接对应一个线程,或者提供一个线程池,无论哪种,性能都不能满足日益增长的请求,吞吐量低.

Reactor 单线程: 一个全局 Selector,一个Reactor 线程.reactor 接收所有连接给 acceptor;acceptor 负责建立 handler 并传递 selector和 channel;handler 负责将 channel 注册到 selector.每个 channel 都有自己的 handler,要么是 acceptor,要么是 handler.缺点是只有一个线程,多核 cpu 利用率不高,大量请求时性能下降,而且线程出错会影响整个服务.

Reactor 多线程: 和 Reactor 单线程不同的地方,就是将原来基于 NIO 事件处理的部分改进为多线程处理.代码结构都一样,只是将读完的数据交给线程池去处理了.改善了性能,但是还是存在单点,依然有风险.

多 Reactor: 将Reactor 多线程的单点部分改成多线程结构,将原来的一个 selector 扩展成多个.

服务端的线程模型和Multithreaded Reactor模型类似.BossGroup是Acceptor,负责处理TCP连接同时将连接注册到Worker中;WorkerGroup是IO读写.客户端线程模型很简单,只有一个WorkerGroup.

NioEventLoop被设计成串行化,防止频繁切换线程和线程冲突带来的开发难度和性能损耗.由于消息读取,解码,业务始终在一个线程中执行,对时间要求可控.复杂业务如外网交互,读写DB等应该由业务线程池统一处理.

  1. I/O 线程 NioEventLoop 从 SocketChannel 中读取数据报,将 ByteBuf 投递到 ChannelPipeline,触发 ChannelRead 事件.
  2. I/O 线程 NioEventLoop 调用 ChannelHandler 链,直到将消息投递到业务线程,然后 I/O 线程返回,继续后续的读写操作.
  3. 业务线程调用 ChannelHandlerContext.write(Object msg) 方法进行消息发送.
  4. 如果是由业务线程发起的写操作,ChannelHandlerInvoker 将发送消息封装成 Task,放入到 I/O 线程 NioEventLoop 的任务队列中,由 NioEventLoop 在循环中统一调度和执行.放入任务队列之后,业务线程返回.
  5. I/O 线程 NioEventLoop 调用 ChannelHandler 链,进行消息发送,处理 Outbound 事件,直到将消息放入发送队列,然后唤醒 Selector.进而执行写操作.

协议设计

公有协议: 如HTTP; 私有协议: 如Thrift. 同一种协议可能有多种序列化形式,如HTTP可以有XML和JSON及Protobuf等.

考虑点: 协议中立,大部分语言都支持; 序列化工具链以及可读性,序列化性能等.

常见的选择有JSON, Protobuf.

Protobuf通过proto文件生成代码,使团队保持协议一致;提供Java支持,Netty也有默认支持;性能好.

长连接实现

TCP协议支持KeepAlived机制,但是默认不打开,心跳间隔是2小时.

使用Netty自带IdleStateHandler,并关注userEventTriggered方法.服务器端进行hb失联次数统计;客户端注意在channelInactive时重连,需要将connect方法独立出来.

  1. 如果系统只有一个服务端 port 需要监听,则 BossGroup 线程组线程数设置为 1
  2. 使用 Netty 的线程池,共三组线程池,分别为 BossGroup,WorkerGroup 和 ExecutorGroup.其中,BossGroup 用于接收客户端的 TCP 连接,WorkerGroup 用于处理 I/O,执行系统 Task 和定时任务,ExecutorGroup 用于处理业务加解密,限流,路由,及将请求转发给后端的服务等业务操作.

  3. 使用EventExecutorGroup另起一个线程池来执行耗时操作,以免堵塞I/O线程.具体见ChannelPipe Javadoc.

  4. LVS提供DR模式,可以将VIP后面的多个NIO服务器与客户端分别保持长连接.

SpringBoot整合要点

  1. 使用IdleStateHandler(int readerIdleTimeSeconds,int writerIdleTimeSeconds,int allIdleTimeSeconds)来管理连接,当读超时,写超时,后者读写任一个超时时会产生一个事件.可以在userEventTriggered方法中实现心跳机制.客户端关注写超时事件,服务器端关注读超时事件.
  2. 集成Actuator或者SpringBootAdminSBA监控.

  3. 通过@Component启动服务.

  4. 客户端多连接实现
