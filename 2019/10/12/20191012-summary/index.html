发表于 2019-10-12 | 更新于 2019-10-28 | 分类于 interview | 阅读次数：

总体感受: 离一个好的架构师还有很长的路,主要是技术视野和实践经验不足.

  • 跟紧主流技术,关注一线大厂动态,多听多看相关分享.
  • 付诸实践,避免细节上把控不好,眼高手低.

技巧: 需要不断dry run, 反问自己.

  1. 精选一个具有代表性的案例,需要不断演练,360无死角反复打破砂锅问到底—为什么要这样,还有没有其他的办法?(案例准备没做好)
  2. 语速张弛有度,有时候需要观察面试官的反应.(整场下来,语速偏快,观察偏少,思考也少,很多答案都是脱口而出即纯经验)
  3. 面试不是聊天,更不是交心,需要话术.回答问题不能随便,不经思考.(个别问题比如缺点,回答未经思考,也没有按照计划内容回答)
  4. 需要换位思考,如果我主持面试,我希望这个位置的候选人有什么特质?(岗位匹配思考)
  5. 表达要清楚,不能带有任何假设.(多说容易犯错,但是潜在假设更不好,还是力求全面,结合第2点)
  6. 基本功要硬.

问题复盘:

  1. 中台

    中台可以简单理解成APaas(领域型Paas,或者说成应用型Paas).就是把业务通用的可共用平台抽出来,进行服务化改造,业务应用可以调用这些服务进行快速开发.字面理解也可以是架构中承上启下的组件,上层有具体应用,下面有iaas或者gpaas(通用性Paas).

    主要目的是业务复用,缩短开发周期.(可以节省成本,抓住时间窗口,多终端适应,技术输出,甚至业务编排即组合各种中台能力)

    典型案例: 用户管理服务(注册,登录,单点登录,管理,鉴权)就可以做成一个中台.

    和常见概念区别:

    • 定制.中台不是定制,会加速定制过程(通过业务编排).
    • 架构.中台是架构的一部分.
    • 中间件.中台除了具有通用平台的能力,还具有业务特征.中间件一般是提供通用能力,无差异支撑各种业务.如数据库中间件.

    SuperCell: 不同的游戏,但是相同的用户系统,支付系统,游戏引擎,底层技术等.”小前台-大中台”.

    具体分类: 业务中台, 技术中台(类似中间件,具有一定业务个性), 数据中台(比如建模,分析,画像等),算法中台(图像识别,推荐算法等).

    适用阶段: 0-1创业公司不适合,最重要的是生存; 1-N复制阶段适合,快速试错.

    隐喻: 厨师(研发), 调料(数据), 糖醋排骨(产品), 糖醋酱汁池子(中台).

  2. ServiceMesh

    微服务架构的核心问题: 服务发现和负载均衡.

    常见模式: 集中代理,客户端嵌入,独立嵌入代理.

    • 集中代理(F5/Nginx): 依赖运维,手工配置,明确规则.客户端通过请求域名,域名指向代理,代理根据根据规则均衡.微服务之前的主流模式.
    • 客户端嵌入(SpringCloud/Dubbo): 服务发现和负载均衡的逻辑都存在客户端的本地库里,比如通过FeignClient底层使用Ribbon方式.主要关注内部服务的治理,而且业务和底层通信耦合在一起了.对于需要互联网访问的服务,还是通过DNS+Nginx的方式来映射扩展.
    • 独立进程嵌入代理: 前两种方式的结合体.由单一进程负责服务发现,负载均衡,熔断限流,安全控制,监控等功能.

    ServiceMesh(服务网格)的几个概念(本质是独立进程代理),还是解决服务发现和通信问题.

    • Side-Car: 业务逻辑和代理在一台主机上,业务是主驾驶,代理是边车.用于服务发现和调用.
    • Control-Plane: 控制平面,是个独立组件,用来集中配置服务之间的关系以及流量挑拨等.
    • Data-Plane: 数据平面,指配置了Sidecar的服务主机.

    Istio实现: ServiceMesh的开源框架,逻辑上也是数据平面和控制平面2部分.具体组件有:

    • Envoy: 就是Sidecar.C++实现.阿里有SOFAMosn用来替代原生的Envoy.
    • Mixer: 负责收集代理采集的度量数据,进行集中监控.
    • Pilot: 为Sidecar提供服务发现,智能路由,弹性(超时,重试,断路器等)流量管理.
    • Citadel: 负责安全控制数据的管理和下发.

    ServiceMesh可能的问题:

    1. 不是新东西,部署代理多,需要规模化部署,以及做好监控.
    2. 落地的实践不多,大厂推广可能有自身的商业利益如Google推Kubernetes.

    其实代理模式1和模式2还可以结合成另一种形式,即在模式1的基础上,将代理和注册中心结合起来,实现代理的自发现功能.

  3. Redis单线程

    Redis是单线程程序,操作的是内存数据库,一般来说程序处理内存数据的速度要远高于网卡接受数据的速度,因此Redis瓶颈在网络IO或者内存上.另一方面Redis使用了非阻塞IO多路复用机制(epoll, kqueue等).

  4. SETNX 和 SET NX EX

    分布式锁的用法应该是SET NX EX, SETNX不能指定EX参数.返回值也不同.

  5. Redis内存清除机制

    采用定期删除和惰性删除策略.Redis是单线程,所以时间长的任务都要特别注意,因此内部不会有遍历所有Key的操作,也不会有长期占用CPU的任务.因此不会定时删除过期key.

    定期删除默认每100ms随机抽查key并决定是否过期.惰性删除就是每次获取key的时候做一个排查.这两种策略会导致内存中过期key越来越多,因此还有内存淘汰策略.

    LRU: 固定长度的链表,默认先进先淘汰,对于高频key,先删除再入队列.Redis采用类似实现:随机取出若干个key,然后按照访问时间排序后,淘汰掉最不经常使用的.

  6. ZK选主,锁,API交互

    ZK只知道理论部分,没有实际的编程经验.

    回来把ZK的api过了一遍,从安装,到命令行,到Java测试.

    再到选主,队列,锁,挨个用代码实现一遍.参考ZK_recipes.

  7. Kafka消费者正确用法

    Kafka的核心功能: 消息系统(队列,发布/订阅)和流处理.

    应用场景: 注重实时消费,比如航班变化,价格变化,用户行为,GPS痕迹等.

    Kafka Consumer API: 直接读取流数据,并对其执行实时操作,例如推送商品促销活动,发送邮件等.

    Kafka Connect Sink API: 读取实时流并将其存储到目标介质中,例如存到HDFS,HBase等.

    ETL: 将数据从孤立的数据源中采集出来(日志,数据库等),进行处理后汇集到一个可以被计算平台高效访问的目的地,此过程被称为ETL,即数据的抽取Extract,转换Transform和加载Load.

    案例: 使用ELK+Kafka来搭建一个日志系统:主要使用spring aop进行日志收集,然后通过kafka将日志发送给logstash,logstash再将日志写入elasticsearch,这样elasticsearch就有了日志数据了,最后使用kibana将存放在elasticsearch中的日志数据显示出来,并且可以做实时的数据图表分析等等.

  8. ELK

    ELK的核心是ElasticSearch.ES如何做到亿级数据查询毫秒级返回呢?

    主要还是用Filesystem cache.往ES里写数据,其实都是写到磁盘文件里了.查询的时候,OS会将文件里的数据自动缓存到cache里.如果cache有更多的内存,可以容纳更多的索引数据,就会有更高的性能.

    至少cache的内存要有数据总量的一半;或者将索引数据控制在一定范围内走内存搜索(例如只存储关键字,剩余信息存到db或者hbase,一般建议es+hbase结构).hbase适合存储海量数据的在线存储,但是不要做复杂查询,可以做简单的id查询或者范围查询,这样整体速度就很快.

    做好ES数据的预热(后台程序干预加载),冷热分离,避免复杂查询(数据做好冗余而不是在ES里关联),避免深度翻页(越深越慢).

    Index相当于库,Type相当于表,Document相当于行.

  9. 设计一个网关

    网关聚合内网多个服务,给前台(多端)提供一个统一入口,涉及功能有: 鉴权,限流,防重/防刷,降级,监控,灰度,协议转换等.
