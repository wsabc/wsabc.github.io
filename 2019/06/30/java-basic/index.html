Java 中最常用的数据类型

最常用的有：String，int，boolean，long，byte。其实也剩不了几个了：short，float，double，char。
总结如下：（参照 Language Basics）

类型 长度（bit） 符号 范围 赋值原子性 默认值
boolean 未精确定义 - true/false 是 false
byte 8 有 [-128, 127] 是 0
short 16 有 [-32768, 32767] 是 0
int 32 有 [-2^31, 2^31-1] 是 0
long 64 有 [-2^63, 2^63-1] 否 0L
float 32 - 精确到小数点后6位 是 0.0f
double 64 - 精确到小数点后15位 否 0.0

或者[0, 65535]		 是 '\u0000'
String - - - - null

除了String上述类型都叫做Primitive Data Type原始数据类型。如果是整数，都是二进制补码格式。如果是小数，都不能用作精确计数（应该使用BigDecimal）。

二进制补码（tow's complement）：第一位是符号位。如果是正数，所有的补位都是0；如果是负数，所有的补位都是1。如0111表示7；1000表示-8。

整数类型提升原则：如果最初的数据类型是有符号的，进行符号扩展（按照符号位扩展）；如果是char，不管被提升成什么类型，都进行零扩展。

精度：float和double在计算机上的存储都遵循IEEE规范，使用二进制科学计数法，都包含三个部分：符号位，指数位和小数部分。float的符号位，指数位，小数部分为1，8，23。double是1，11，52。

Java中整数都是有符号类型，从JDK8开始支持无符号类型。JDK8之前如果需要某个宽度的无符号类型，可以用>>>这个是无符号右移操作符，或者使用下一个宽度的带符号类型来模拟，如用int模拟无符号short。

JVM字长：CPU字长是指CPU同时参与运算的二进制位数，现在主流都是64位。OS字长要看OS的位数，跟机器字长不一定一致，如64位机器上装32位的系统。JVM的字长没有明确规定，但是必须能够支持byte、short、int、char、float、returnAddress和reference中的最大者，两个字长足以持有long和double。因此可以看出JVM字长是32位。

字面量（literal）

原始类型实例化不需要new关键字。字面量就是某个固定值的源代码直接表示。整型字面量默认是int;浮点型字面量默认是double。整型字面量有三种表现形式：十进制；十六进制0x和二进制0b。

自动转换：支持窄类型到宽类型的自动转换，如将short赋值给int。同时也支持宽类型到窄类型的自动转换，但是前提是不能超过窄类型的范围。注意提升成float或者double时可能会出现精度损失。注意：float到double不能自动提升（因为存储格式不同）。

强制转换：把宽类型转成窄类型，或者同宽度之间转换（如short和char），就需要强制转换。注意：字面量不需要强制转换。

类型提升：除了赋值以外，表达式计算过程中也可能发生一些类型转换。类型提升规则有：整型默认int，short/byte/char都会提成int；如果操作数有long，就都会提升成long。float和double同样适用。

赋值（Assignment）的原子性

  1. 所有primitive类型的赋值都是原子性的。long和double除外。
  2. 带有volatile修饰的long和double的赋值是原子性的。

因为对于普通64位长度的long和double，Java会先写入32bits，然后再写32bits。这会导致一个线程看见A写的前32位，然后又看见B写的后32位。这个JVM规范定义的。

注：引用类型读写操作也总是原子性的，无论他们被实现成32位还是64位。

数组

特点：引用类型；单一类型元素；固定长度；通常内存连续（JVM没要求），按索引访问。

堆中的对象要么是一个类的实例，要么就是一个数组。

Java 异常分类及设计

Throwable有两个子类Error和Exeption。Exception下面又分Checked-Exception和Runtime-Exception。

Error表示的是应用程序无能为力的场景，无法参与也无法恢复。如JVM崩溃，或者OOM。

Checked-Exception表示程序可以参与并尝试恢复的场景，如文件找不到。

Runtime-Exception表示由于坏的编程习惯导致的运行时异常，如空指针，数组越界等。

设计原则

  1. 使用合适的，准确的异常，提供精准的错误消息。
  2. 尽早失败，如果不能处理异常就应该尽快抛出去。
  3. 记录异常信息，但是防止多个层面重复记录。
  4. 不要用异常来控制流程。
  5. 对于运行时异常，需要在抛给用户前做统一处理，避免泄露过多细节，也不能抛一个用户无法处理的异常。
  6. 业务异常应该都是Checked-Exception，返回给用户让用户处理。

Vector vs. ArrayList

接口相同 List，RandomAccess List，RandomAccess
底层数据结构相同 Object[] elementData transient Object[] elementData
都是可变长度，add时都做扩容校验 elementCount size
都需要复制数组 Arrays.copyOf Arrays.copyOf

线程安全不同 安全，方法同步（同步容器） 不安全

同步容器：对容器的方法添加Synchronized提供同步能力的容器。如Vector。类似的还有Collections.synchronizedXXX也是返回一个同步容器。开销略大。同步容器的单个操作是线程安全的，但是复合操作需要外部额外加锁控制。

并发容器：相对同步容器来说，避免多线程串行执行容器方法，提高并发性的容器，如ConcurrentHashMap等。尽量避免锁，提供一些线程安全的符合操作。

HashSet

HashSet特点是不允许重复元素，但是允许null存在。底层是一个HahMap。HashMap保证了key不会重复，重复即覆盖。

HashMap有的问题HashSet也有。比如环形链表（hash相同，key不同，扩容形成环形链表）。

LinkedList

实现了AbstractSequentialList,List和Deque接口。底层是一个双向链表。add方法通过移动尾节点指针来插入数据（内存不连续，分散存储），效率比ArrayList高很多（没有数组复制的过程）。get方法通过判断index和头部以及尾部的距离判断是从头部开始迭代还是从尾部开始查找（index < (size >> 1)），时间复杂度O(n/2)，查找效率低下，不如ArrayList（按照索引查找）。线程不安全。

LinkedHashMap

由于HashMap迭代出来是无序的（写入的时候是根据key的hash映射到数组中），LinkedHashMap提供了一个有序的HashMap。底层基于HashMap实现，扩展了HashMap的HashMap.Entry<K,V>使之变成了一个双向链表来保证顺序性（通过添加header和tail标记以及before，after指针）。有2种顺序：插入顺序（默认）和访问顺序（调用一次get就将当前元素移到队尾）。线程不安全。

LinkedHashMap vs. TreeMap

相同点：都是继承Map接口；都是线程不安全。

不同点：底层结构不同（LHM是数组+链表+红黑树，TM是红黑树）；排序方式不同（LHM是插入顺序和访问顺序；TM是根据Key的比较器排序）。
