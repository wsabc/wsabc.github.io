现代CPU

现代 CPU 都是从晶圆(wafer)加工成晶粒(die),再做成 CPU 的.在晶圆到晶粒的制作过程中,良品率跟晶粒的大小有关系.晶粒越小良品率越高.根据晶粒的大小,有以下几种布局:

  1. 单块 CPU 单核心: 无论晶粒大小,一个晶粒一个 CPU 核心一个物理块.这种性能有限.可以多线程并发执行,使用时间片轮换或者优先级策略.有上下文切换的开销.
  2. 单块 CPU 多核心: 根据晶粒的大小还可以分为: 一个大晶粒上多个 CPU 核心;多个小晶粒,每个晶粒一个 CPU 核心.共享一个大晶粒的性能要好,但是成本高.物理上还是一个模块.可以多线程并行执行.每个核心上还是并发执行.
  3. 多个CPU (单/多)核心: 物理上多块 CPU,每个 CPU 都可以是单核,或者多核.

现在主流的 Intel 就是大晶粒多核; AMD有多个小晶粒多核.举个例子形象地解释它们的区别:

大地产商建造房子都是通透大平层(大晶粒),平层上各个商家(CPU 核心)很方便沟通,也不太堵塞.小地产商没有能力建大平层,建几个小高楼(多个小晶粒),通过地下通道互联,容易堵死.再小的地产商连地下通道也开发不了,只能远远的建立几个小楼,通过公交等串门互联.性能最差.

多线程核心问题

  1. 上下文切换. 在具体的 CPU 上多个线程轮换时间片,需要中断时保存现场(程序计数器),继续时恢复现场,这就是上下文切换.这是一个耗时操作.可以通过vmstat命令查看上下文切换次数.
  2. 资源限制. 线程不是越多越好.应该根据任务的特性决定线程的数量.性能和线程数量不是正比关系(切换有开销).
  3. 资源冲突: 多个线程访问同一个资源时就有竞争问题.比如死锁问题.通常一个线程一个锁,访问一个资源.要保证锁的释放.

影响多线程正确性的几个因素:

  1. 原子性操作: 识别非原子性操作.保证原子性的几个常见手段: synchronized, lock和cas.
  2. 内存可见性: 因为工作内存和主内存的设计,多线程之间内存有一致性问题.volatile用于保证可见性.
  3. 指令顺序性: 防止 JVM 优化指令导致可见性不一致.具体参照各种happens-before规则.

锁

锁的特性: 重入性; 排他性; 阻塞性.

  • 重入性: 同一个线程可以反复进入,通过计数器来记录,知道计数器为 0 时锁释放.如ReentrantLock.
  • 排他性: 一个线程拿到了锁,其他线程就需要等待,直到收到锁释放通知,重新竞争.比如synchronized.
  • 阻塞性: 一旦线程竞争不到锁,就会放到阻塞队列里等待,而不是失败返回.由竞争者决定.

针对于锁和当前进程的关系,可以分为:

  • 同一进程内: 比如一个 Java 程序里的 N 个线程,他们可能竞争一个锁.
  • 不同进程间: 也就是分布式锁.不同的进程或者线程去竞争同一个分布式锁.如 Redis 分布式锁,数据库记录等.

数据库锁的弊端

利用唯一索引,可以通过判断往数据库插入记录是否成功来判定谁获得分布式锁.使用完毕删除记录即释放锁.缺点如下:

  1. 数据库可能是单点 -> 主从结构
  2. 没有重入性,释放锁之前无法再次进入了(数据库记录已经有了) -> 仿照重入锁增加计数字段
  3. 没有阻塞性,获取不到锁的线程直接失败退出了,无法阻塞等待,只能下次重新获取 -> 可以循环等待,超时退出
  4. 容易死锁,如果解锁失败以后就没有其他线程能获取到锁了 -> 旁挂线程定期删除未解锁的记录

常见锁

  1. 乐观锁: 广义锁.数据库和 Java 里都有这个概念.Java 常见的是 CAS 实现.参考 AtomicInteger.数据库通常是 version 字段比较.
  2. 悲观锁: 广义锁.数据库里select...for update.Java 里synchronized和 lock 都是.
  3. 公平锁: 指多个线程按照申请锁的顺序来获取锁,线程直接进入队列中排队,队列中的第一个线程才能获得锁.公平锁的优点是等待锁的线程不会饿死,缺点是整体吞吐效率相对非公平锁要低,等待队列中除第一个线程以外的所有线程都会阻塞,CPU唤醒阻塞线程的开销比非公平锁大.
  4. 非公平锁: 多个线程加锁时直接尝试获取锁,获取不到才会到等待队列的队尾等待.但如果此时锁刚好可用,那么这个线程可以无需阻塞直接获取到锁,所以非公平锁有可能出现后申请锁的线程先获取锁的场景.非公平锁的优点是可以减少唤起线程的开销,整体的吞吐效率高,因为线程有几率不阻塞直接获得锁,CPU不必唤醒所有线程.缺点是处于等待队列中的线程可能会饿死,或者等很久才会获得锁.
  5. 无锁 - 偏向锁 - 轻量锁 - 重量锁: 这 4 个是指锁的状态,专门针对synchronized的.锁状态从低到高,只能升级不能降级.无锁没有对资源进行锁定,所有的线程都能访问并修改同一个资源,但同时只有一个线程能修改成功.偏向锁是指一段同步代码一直被一个线程所访问,那么该线程会自动获取锁,降低获取锁的代价.偏向锁只有遇到其他线程尝试竞争偏向锁时,持有偏向锁的线程才会释放锁,线程不会主动释放偏向锁.偏向锁的撤销,需要等待全局安全点,它会首先暂停拥有偏向锁的线程,判断锁对象是否处于被锁定状态,撤销偏向锁后恢复到无锁或轻量级锁的状态.当锁是偏向锁的时候,被另外的线程所访问,偏向锁就会升级为轻量级锁,其他线程会通过自旋的形式尝试获取锁,不会阻塞,从而提高性能.若当前只有一个等待线程,则该线程通过自旋进行等待.但是当自旋超过一定的次数,或者一个线程在持有锁,一个在自旋,又有第三个来访时,轻量级锁升级为重量级锁.重量级锁依赖于操作系统互斥锁实现(“重量”的原因).
  6. 适应性自旋: 阻塞或唤醒一个Java线程需要操作系统切换CPU状态来完成,这种状态转换需要耗费的处理器时间有可能比用户代码执行的时间还要长(如果同步代码块中的内容过于简单的话).而实际上在许多场景中,同步资源的锁定时间都很短,为了这一小段时间去切换线程会让系统得不偿失.如果物理机器有多个处理器,能够让两个或以上的线程同时并行执行,我们就可以让后面那个请求锁的线程不放弃CPU的执行时间,看看持有锁的线程是否很快就会释放锁.这样做的好处就是当前线程可能可以不必阻塞而是直接获取同步资源,从而避免切换线程的开销.自旋锁不能代替阻塞,会消耗 CPU 时间.适合锁占用时间短的场合.使用-XX:PreBlockSpin可以限定自旋的次数.超过次数没有获取锁就挂起线程.自旋锁在JDK1.4.2中引入,使用-XX:+UseSpinning来开启,JDK 6中变为默认开启,并且引入了自适应的自旋锁.在自旋锁中另有三种常见的锁形式:TicketLock, CLHlock和MCSlock.
  7. 重入锁: 重入锁又名递归锁,是指在同一个线程在外层方法获取锁的时候,再进入该线程的内层方法会自动获取锁(前提锁对象得是同一个),不会因为之前已经获取过还没释放而阻塞.ReentrantLock和ynchronized都是可重入锁,可重入锁的一个优点是可一定程度避免死锁(否则递归调用就死锁了).
  8. 读写锁(共享锁和独占锁): 独享锁也叫排他锁,是指该锁一次只能被一个线程所持有.如果线程T对数据A加上排它锁后,则其他线程不能再对A加任何类型的锁.获得排它锁的线程即能读数据又能修改数据.synchronized和Lock的实现类就是互斥锁.共享锁是指该锁可被多个线程所持有,如果线程T对数据A加上共享锁后,则其他线程只能对A再加共享锁,不能加排它锁.获得共享锁的线程只能读数据,不能修改数据.读写锁实现读读的过程共享,而读写,写读,写写的过程互斥.
  9. 分布式锁: 分布式场景中由其他进程提供一个锁,通常是互斥锁,重入锁.

参考美团技术文章.

无锁编程

  1. 无锁编程: 将数据分片(如 hash 取模),每个线程负责自己的片,没有竞争.
  2. CAS(compare and swap): 如 Atomic 包.
  3. 合理创建线程,避免大量 waiting 状态的线程(避免上下文切换).

CAS

CAS是一条CPU的原子指令(cmpxchg),不会造成数据不一致问题.Unsafe提供的CAS方法(如compareAndSwapXXX）底层实现即为CPU指令cmpxchg.

CAS原语有三个参数:内存地址,原值,新值.如果内存地址的值等于原值(没有发生更新),此时可以修改成新值;否则表示修改失败,由用户决定后续操作(报错还是重试).

ABA 问题

CAS需要在更新值的时候检查内存值是否发生变化,没有发生变化才会更新内存值.但是如果内存值原来是A,后来变成了B,然后又变成了A,那么CAS进行检查时会发现值没有发生变化,但是实际上是有变化的.ABA问题的解决思路就是在变量前面添加版本号,每次变量更新的时候都把版本号加1,这样变化过程就从A－B－A变成了A1－B2－A3了.

JDK从1.5开始提供了AtomicStampedReference类来解决ABA问题,具体操作封装在compareAndSet()中.首先检查当前引用和当前标志与预期引用和预期标志是否相等,如果都相等,则以原子方式将引用值和标志的值设置为给定的更新值.

循环时间长开销大

CAS操作如果长时间不成功,会导致其一直自旋,给CPU带来非常大的开销.

只能保证一个共享变量的原子操作

对一个共享变量执行操作时,CAS能够保证原子操作,但是对多个共享变量操作时,CAS是无法保证操作的原子性的.

Java从1.5开始JDK提供了AtomicReference类来保证引用对象之间的原子性,可以把多个变量放在一个对象里来进行CAS操作.

ThreadLocal

问题: 在代码A处设置一个值,经过重重方法调用之后,在代码B处取出来使用.

方案: 逐步修改方法签名 – 不可取,修改繁琐,而且有些代码不能修改(比如采用了第三方的库); 使用静态方法存取 – 不可取,在多线程的环境下容易产生干扰,值互相覆盖,不是线程安全的操作. 那有没有类似在Thread对象中设置一个属性的办法呢?

这个就是ThreadLocal类的初衷. Thread类中有一个ThreadLocalMap用于存放线程级别的变量,委托给ThreadLocal类操作.之所以是Map是因为可能会存在多个变量(变量就是key,值就是value).

原理: 每个线程中都有一个ThreadLocalMap数据结构,当执行set方法时,其值是保存在当前线程的threadLocals变量中,当执行get方法中,也是从当前线程的threadLocals变量获取的,保证线程之间不会干扰.ThreadLocalMap是个普通类,跟HashMap无关,底层也是一个Entry[],但是没有链表.发生hash冲突的时候就寻找下一个空位.

好处:

  1. 在多线程环境下,可以防止自己的变量被其它线程篡改.
  2. 在父子线程的环境中,可以通过InheritableThreadLocal传递父线程的变量.

坏处:

  1. 发生线程切换的时候,需要手动传递变量.比如业务代码中新开一个线程.
  2. 可能发生内存泄漏.
