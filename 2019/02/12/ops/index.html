运维

发表于 2019-02-12 | 更新于 2019-09-30 | 分类于 ops | 阅读次数：

运维是一个综合性岗位,涉及网络,系统,安全,应用架构,存储,开发等多个技术领域.常见的划分角度有:

  1. 按照领域划分: 基础设施运维, 系统运维, 数据运维, 应用运维, 云平台运维, 容器运维.

  2. 按照技术面划分: 安全运维, 性能运维, 数据运维, 集成运维.

  3. 按照流程划分: 构建/持续集成,发布; 安装,部署/升级/迁移/合并/扩展; 配置/初始化/配置升级; 备份/恢复/传输; 日志,监控,预警; 诊断排查,优化.

基本的运维技能

  1. Linux基础知识: shell, yum/rpm安装, 系统服务.
  2. 软件服务: HTTP服务器, FTP, DNS等.
  3. 系统管理: 进程, 内存, 磁盘, 权限等.
  4. 监控: Zabbix, Nagios等.
  5. 云计算/虚拟化
  6. 脚本编程: Python, Shell, Perl等.

Web运维

  1. Web基础: Tomcat, Nginx, TCP/HTTP
  2. 数据库: MySQL, PostgreSQL, Oracle, Redis, MongoDB, 主从复制, 读写分离
  3. 安全: DDos, Iptables, web shell等
  4. 集群: LVS, KeepAlived, Nginx负载均衡
  5. 存储: OSS, NFS
  6. 持续集成: git, jenkins, docker
  7. 云平台: 阿里云, 腾讯云

设计运维友好的系统

在运维工作中,自动化程度是一个很重要的指标.运维友好(operations friendly)则是尽量减少人工干预,如果一个系统很少需要人工干预,能够自动检测并恢复服务,只有极少数复杂故障才需要人员介入,那这个系统就是运维友好的.

  1. 预期失败: 任何一个组件在任何一个时间都可能失败,也可能会影响跟它相关联的系统.要预见并能处理.
  2. 保持简单: 复杂容易产生更多问题,简单容易做正确.
  3. 尽量自动化: 人会犯错,但是程序可以量化并被相信.

根据上面三个基本原则,可以在具体实践中逐步细化.

  • 注意系统的冗余和故障恢复.列出可能的故障组合,确保在不影响业务的前提下能继续运作.
  • 升级软件注意避免重大用户体验的改变.
  • 服务健康状况快速检查.
  • 能产生具体的性能数据,健康数据,以及改动的审计日志.
  • 实现降级和限流.
常见性能指标
  • 响应时间RT: 系统对请求做出响应的时间.一般来说指整个系统所有功能的平均响应时间或所有功能的最大响应时间.与用户的感觉密切相关.RT快慢也因为具体的场景不同标准也不同(游戏要求快,编译要求不高等).
  • 吞吐量Throughput:系统在单位时间内处理请求的数量.
  • 并发数: 系统可以承载的正常使用系统功能的请求数量.并发数并不准确,比如并发用户数有时分注册用户数,有时是在线用户数,有时是同时发请求用户数.
  • 每秒查询数量QPS: QPS=并发数/RT.类似的还有TPS(每秒事务数), HPS(每秒HTTP请求数).这些都是吞吐量的量化指标.
  • PV(PageView):页面浏览量,一个页面用户每一次访问记录1次,用户对同一页面的多次刷新,累计次数.
  • UV(UniqueVistor):独立访客数,统计1天内某站点的访问用户数(以cookie为依据),一台电脑终端为一个访客.类似的还有RV(RepeatVistors)重复访问者数量.
  • 独立IP数:指1天内多少个独立的IP浏览了某页面.同一个路由器上的用户被认为是一个IP.
常见监控手段
  • SNMP: 简单网络管理协议, 涉及三个参与者:被管理者(Devices),管理者(NMS),代理(Agent).管理者通过UDP协议和代理交互,实现读(GET),写(SET)和通报(Trap).占用161和162端口.每个管理对象都有自己的OID,自己状态和统计信息都在MIB(管理信息库)中.使用SNMP4J类库.
  • SpringBoot Actuator:对SpringBoot应用的监控和管理.常用的有/health,/metric,/trace等接入点.

工具

  • SED: 文本流编辑器.
